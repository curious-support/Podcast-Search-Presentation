Listen: Podcast Discovery

Every podcast publishes an RSS feed. We use it as our single entry point to discover and pull every episode automatically.

  1. Point the engine at any RSS feed URL
  2. It parses the feed and discovers every episode
  3. For each episode, it extracts:
    title, audio URL, podcast name, artwork
  4. Works with any podcast — Spotify, Apple, independent

What is RSS?

Think of RSS as a structured menu for podcasts. One URL gives you a machine-readable list of every episode — title, audio file, artwork, duration — all in one place.

podcast_feed.rss

Listen: Audio → Text

We don't download audio files. The URL goes straight to a cloud transcription service — it comes back as text with speaker labels and timestamps.

  1. Audio URL sent directly to Deepgram — zero file downloads
  2. Cloud-to-cloud: podcast CDN → Deepgram → our database
  3. Speaker diarization identifies who is talking
  4. Handles episodes up to 3+ hours long

Index: Chunking

A raw transcript is too long to search as-is. Chunking breaks it into bite-sized pieces that each carry a single topic.

  • Groups transcript segments into ~1000-character chunks
  • Preserves speaker labels within each chunk
  • Tracks start/end timestamp range per chunk
transcribe.py — _chunk_segments
1def _chunk_segments(self, segments, target_size=1000): 2 chunks = [] 3 4 for seg in segments: 5 speaker = seg.get("speaker") 6 line = f"[Speaker {speaker}] {seg['text']}" 7 8 if current_len + len(line) > target_size: 9 chunks.append({ 10 "text": "\n".join(current_texts), 11 "start_time": current_start, 12 "end_time": current_end, 13 "speakers": sorted(current_speakers), 14 }) 15 current_texts, current_len = [], 0 16 17 current_texts.append(line) 18 current_len += len(line) 19 20 return chunks

Why We Need Chunking

A 1-hour podcast is ~10,000 words. You can't embed that into a single meaningful vector.

  1. Embedding models have input limits — an entire transcript won't fit into one vector
  2. One vector for a whole episode captures everything but matches nothing specific
  3. Smaller chunks = sharper meaning — each chunk has one focused topic search can find
  4. Each chunk becomes a searchable unit with its own timestamp, speakers, and topic

Index: Text to Vectors

Each chunk is converted into a 1024-dimensional vector — a numerical fingerprint of its meaning — and stored alongside rich metadata.

  1. Each chunk is embedded using Pinecone's llama-text-embed-v2 model
  2. Stored with rich metadata: episode, podcast, timestamps, speakers
  3. Batched upserts for efficient writes
  4. Currently indexing 674 episodes across 9 podcasts

Why a Vector Database?

A regular database matches exact keywords. A vector database understands meaning — even when the words are completely different.

We use
Regular Database Vector Database
Query WHERE text LIKE '%hire PM%' Finds closest meaning to "how to hire a PM"
Result Only exact keyword matches Also matches "recruiting product managers"

Search: Query Embedding

When you search, your question is converted into the same vector space as the chunks — so we compare meaning, not just keywords.

Response: Similarity Search

The query vector is compared against every stored chunk. The closest matches become context for the LLM to write a grounded answer.

  • Pinecone returns the most relevant chunks ranked by similarity score
  • Low-relevance results are filtered out (score < 0.20)
  • Matching chunks are passed as context to GPT-4o-mini to generate a cited answer
